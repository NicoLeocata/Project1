CSS Frameworks

Bootstrap

standalone

Description: Mature CSS framework focused on responsive components and utilities.

Technology: CSS, optional JS (vanilla/bootstrap bundle).

Learning curve: Low — quick to prototype with classes.

Use case: Rapid prototypes, admin UIs, consistent layout systems.

Bootstrap is my go-to for fast scaffolding when visual consistency and speed matter more than bespoke styles.

Tailwind CSS

nodejs

Description: Utility-first CSS framework for composing custom designs quickly.

Technology: PostCSS build step, Node.js tooling.

Learning curve: Medium — mental model differs from semantic class naming.

Use case: Highly-custom, maintainable design systems and component-based UIs.

Tailwind speeds design iteration but requires a build step; I use it for portfolio pieces where polish matters.

JavaScript Libraries

React

nodejs

Description: Component-driven UI library for building interactive user interfaces.

Technology: JavaScript/JSX, Node.js build tools (Vite, Webpack).

Learning curve: Medium to high depending on ecosystem adoption.

Use case: Complex single-page apps and interactive components.

I pick React when I need predictable state and a rich component ecosystem; otherwise it's overkill for static pages.

GSAP

standalone

Description: High-performance animation library for precise, timeline-driven animations.

Technology: Vanilla JS library, works in browsers directly.

Learning curve: Low to medium — API is straightforward for basic animations.

Use case: Micro-interactions, page transitions, and polished motion design.

GSAP is excellent for smooth, complex animations; I use it sparingly to enhance UX without hurting performance.

Static Site Generators

Hugo

binary

Description: Ultra-fast static site generator written in Go.

Technology: Go binary, templates, front matter content files.

Learning curve: Low to medium — fast builds and simple templating.

Use case: Blogs, documentation sites, and large static sites that need quick builds.

Hugo's speed is compelling for large content sites; I use it when build time and simplicity are priorities.

Jekyll

ruby

Description: The classic Ruby-based static site generator powering many GitHub Pages sites.

Technology: Ruby, Liquid templating, front matter.

Learning curve: Low for basic use; Ruby knowledge helps for advanced customization.

Use case: Personal blogs and simple content sites, especially with GitHub Pages.

Jekyll is dependable for simple blogs; I appreciate its convention-over-configuration approach for straightforward content sites.

SaaS / CMS Tools

Netlify CMS

saas

Description: Git-based CMS providing an editorial UI for static sites.

Technology: JavaScript admin UI that connects to Git workflows.

Learning curve: Low to medium — straightforward setup with Git-backed workflows.

Use case: Teams who want a CMS for static sites without a complex backend.

Netlify CMS brings content editing to static sites with minimal friction; I like it for collaborative blog workflows.

Hygraph

saas

Description: Headless CMS (GraphQL-first) that manages content and delivers via APIs.

Technology: Cloud SaaS, GraphQL APIs, integrates with any frontend.

Learning curve: Medium — requires understanding GraphQL and content modeling.

Use case: Complex content models for multi-platform publishing and decoupled architectures.

Hygraph is powerful for structured content across channels; I use it when the project needs a robust API and editorial roles.

Critical Reflection

Designing with Constraints

After exploring and comparing all these tools, I realized that understanding them helps me appreciate how design and development connect, but it also made me aware of how complex web development can get. Between build steps, dependencies, and frameworks that change every year, it often feels like a world that demands constant technical focus rather than creative direction.

Balancing Act: Creativity vs. Technicality

As a designer, what truly interests me is how design communicates—not how every piece of code works underneath. I enjoy experimenting with tools like Tailwind or GSAP to understand possibilities, but building an entire portfolio from scratch feels unnecessarily heavy for what I want to express.

Using these tools in my portfolio

That's why I'll most likely continue using Framer to design and publish my portfolio. It lets me focus on composition, animation, and storytelling without getting lost in build processes or debugging. It's visual, flexible, and gives me creative control—the parts I care about most.

Reflecting on my learning journey

In the end, knowing about these technologies still shapes how I design interfaces and think about structure. Even if I don't code everything myself, I can collaborate better with developers and make more informed design decisions.
